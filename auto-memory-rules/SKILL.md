# Auto-Memory-Rules 核心执行准则

> 本文件是记忆管理的"唯一真理来源"。

## 触发场景

当用户使用以下关键词或表达意图时，立即启动记忆系统：

### 明确记忆请求
- 🎯 "记住这个"、"记录下来"、"别忘了"
- 🎯 "下次注意"、"以后要..."、"形成规则"
- 🎯 "保存这个偏好"、"记住我的习惯"
- 🎯 "生成规范"、"创建 checklist"

### 纠错和修改
- ⚠️ "不对"、"错了"、"应该是..."
- ⚠️ "改成..."、"换成..."、"不要..."
- ⚠️ "还要..."、"漏了..."、"别这样..."

### 偏好表达
- 💡 "我更喜欢..."、"最好..."、"建议..."
- 💡 "我的习惯是..."、"我通常..."

**重要**：即使用户没有明确说"记住"，当检测到重复的纠错模式时，也应主动询问是否要记录为规则。

## 核心价值

**自动学习用户偏好，形成个性化的代码规范 checklist**

本系统通过一系列脚本自动化记忆管理流程：

### 可用脚本

| 脚本 | 用途 | 何时调用 |
|------|------|---------|
| `generate-integration-rules.sh` | 生成集成规则到编辑器配置目录 | 首次安装或更新集成规则时 |
| `generate-memory-artifacts.sh` | 同时生成索引和清单 | 创建新记忆后（必须） |
| `generate-rules.sh <editor>` | 生成分类规则到编辑器 | 需要更新编辑器规则时（可选） |

### 脚本调用时机

```
用户请求记忆
    ↓
AI 创建记忆文件 (memory/YYYY-MM-DD-xxx.md)
    ↓
调用 generate-memory-artifacts.sh  # 同时生成索引和清单
    ↓
(可选) 调用 generate-rules.sh      # 更新编辑器规则
```

## 最高优先级原则

**用户的实时要求 > 历史记忆**

- 当用户的当前要求与历史记忆冲突时，**无条件执行用户要求**
- 将冲突视为"记忆更新信号"
- 在对话结束时，**询问用户**是否要覆盖旧记忆
- 记忆是参考，不是约束

**示例场景**：
```
历史记忆：API URL 不使用尾部斜杠
用户要求：这个 API 需要带尾部斜杠

AI 行为：
1. ✅ 立即执行用户要求（带尾部斜杠）
2. 💬 提示："注意到这与历史记忆冲突，已按您的要求执行"
3. ❓ 对话结束时询问："是否要更新记忆规则？"
```

## 任务生命周期

### 0. 对话内缓存

在同一对话中维护以下状态以减少 IO：
- `last_keywords` - 上次的关键词列表
- `last_topic_fingerprint` - 话题指纹（前 3 个关键词）
- `loaded_rules` - 本对话已读取的规则文件
- `pending_fragments` - 待提纯的碎片列表

### 1. 对话开始（Pre-task）

**主动检测记忆请求**：
- 扫描用户消息中的触发关键词
- 如果检测到记忆相关请求，立即响应："我会记住这个规则"
- 如果是首次使用，自动初始化记忆系统

1. **检查目录**
   ```bash
   ls -la .kiro/skills/auto-memory-rules/memory/ 2>/dev/null
   ```

2. **自愈初始化**（如果目录不存在）
   - 创建 `memory/` 目录结构
   - 创建 `memory/index.md` 索引文件

3. **加载相关记忆**
   - 根据用户请求的关键词，搜索相关记忆
   - 将匹配的记忆注入到当前上下文
   - 如果没有找到相关记忆，明确说明"未找到相关历史记忆"

### 2. 对话执行（Execution）- 核心功能

**触发条件：**

| 意图类型 | 触发信号 | 优先级 | 处理策略 |
|---------|---------|--------|---------|
| **明确纠错** | "不对"、"错了"、"应该是..." | 🔴 高 | 立即反思 + 记录 |
| **要求修改** | "改成..."、"换成..."、"不要..." | 🔴 高 | 立即反思 + 记录 |
| **偏好表达** | "我更喜欢..."、"最好..."、"建议..." | 🟡 中 | 反思后记录 |
| **方法指导** | "用...命令"、"去...查询" | 🟡 中 | 反思后记录 |
| **交互优化** | 对回答方式、格式、风格的调整 | 🟡 中 | 反思后记录 |
| **一次性修改** | "这里特殊处理"、"临时..." | ⚪ 低 | **不记录** |

**自我反思流程：**

当用户提出任何修改要求时，AI 必须进行三层反思：

1. **表层反思**：用户说了什么？
   - 直接理解用户的字面要求

2. **深层反思**：为什么用户会这么说？
   - 我的回答哪里不够好？
   - 我遗漏了什么重要信息？
   - 我的方案有什么不合理的地方？
   - 我是否理解了用户的真实意图？

3. **原则提取**：这背后的通用原则是什么？
   - 是代码质量问题？（quality）
   - 是用户偏好？（convention）
   - 是最佳实践？（pattern）
   - 是项目规范？（workflow）
   - 是否可以泛化到类似场景？

**执行动作：**
1. **立即反思** - 理解用户要求的深层原因
2. **提取原则** - 识别可复用的通用规则
3. **创建记忆** - 记录到 `memory/` 目录
4. **应用改进** - 在当前任务中立即应用
5. **主动提示** - 告知用户："我已记录这条记忆，下次会自动应用"

### 主动记忆提示

当检测到以下情况时，主动询问用户是否要记录为记忆：

1. **重复纠错**（同一问题被纠正 2 次以上）
   - "我注意到这是第二次提到这个问题，要记录为记忆吗？"

2. **明确的偏好表达**
   - "这看起来是您的编码偏好，要保存下来吗？"

3. **重要的最佳实践**
   - "这是个很好的实践，要形成记忆以便后续自动应用吗？"

**提示格式**：
```
💡 检测到可记录的记忆：[简短描述]
是否要记录到记忆系统？这样下次我会自动应用。[Y/n]
```

### 3. 对话结束（Settlement）

**话题切换判断：**

当出现以下任一条件，视为话题切换，触发结算：
- 明确转折词：「另外」「改成」「换成」「再来」「顺便」
- 本回合关键词与 `last_keywords` 差异 >= 40%
- 用户明确要求新增/修改规则
- 对话自然结束

**结算流程：**

1. **统计本次创建的记忆**
   ```bash
   # 检查今天创建的记忆文件
   ls -la memory/$(date +%Y-%m-%d)-*.md 2>/dev/null | wc -l
   ```

2. **调用脚本更新索引和清单**
   
   **必须调用**（如果创建了新记忆）：
   ```bash
   # 同时生成索引和清单
   bash scripts/generate-memory-artifacts.sh
   ```

3. **可选：更新编辑器规则**
   
   询问用户是否要立即更新编辑器规则：
   ```bash
   # Kiro 用户
   bash scripts/generate-rules.sh kiro
   
   # Frieren 用户
   bash scripts/generate-rules.sh frieren
   
   # Cursor 用户
   bash scripts/generate-rules.sh cursor
   
   # Windsurf 用户
   bash scripts/generate-rules.sh windsurf
   ```

4. **输出结算报告**
   ```
   ✅ 记忆系统结算完成
   
   本次对话记录：
   - 新增记忆：3 条
   - 更新索引：memory/index.md
   - 更新清单：memory/CHECKLIST.md
   
   💡 提示：运行以下命令更新编辑器规则
   bash scripts/generate-rules.sh <editor>
   ```

**提纯判断标准：**
- ✅ 用户明确表达的偏好
- ✅ 重复出现的修改模式（2次以上）
- ✅ 可以抽象为通用规则
- ❌ 一次性的特殊情况
- ❌ 项目特定的实现细节

**质量门控 Checklist：**

核心问题：**这个规则下次能让用户少提修改要求吗？**

应该记录：
- ✅ 用户明确的偏好表达（"我更喜欢..."、"最好..."）
- ✅ 重复出现的修改模式（2次以上）
- ✅ 可抽象为通用原则（跨项目可用）
- ✅ 高成本的错误与修正路径
- ✅ 关键参数/设定/前置条件
- ✅ 用户对交互方式、回答格式的偏好
- ✅ 代码风格、架构模式的偏好

不应记录：
- ❌ 一次性的特殊处理
- ❌ 项目特定的业务逻辑
- ❌ 临时的解决方案
- ❌ 一问一答、没有可重用流程
- ❌ 纯概念解释（没有具体做法）

质量标准：
- [ ] **去具体化** - 移除文件路径、行号、具体变量名
- [ ] **高可执行性** - 规则必须是 AI 读完后能立刻执行的指令
- [ ] **跨任务通用** - 该规则在其他类似场景中依然有效
- [ ] **抗衰减** - 规则逻辑清晰，不会随时间推移而失效
- [ ] **理解原因** - 记录了"为什么"，不只是"做什么"

**冲突处理：**
- **规则演进** - 若用户明确推翻旧习惯，询问后覆盖旧规则
- **场景分支** - 若两个规则在不同场景下均有效，则合并并详细描述各自的触发场景

## 标准 Category 分类

创建 rule 文件时，必须从以下 category 中选择**唯一一个**：

| Category | 中文名称 | 适用场景 | 示例 |
|----------|---------|---------|------|
| `api` | API 相关 | API 调用、接口定义、HTTP 请求、URL 规范 | API URL 尾部斜杠、工厂方法 |
| `types` | 类型定义 | TypeScript 类型、接口、泛型、类型工具 | 查询参数类型、类型适配器 |
| `component` | 组件开发 | React 组件、JSX、组件设计、Props | 组件位置规则、Props 必选性 |
| `hook` | Hook 开发 | 自定义 Hook、useXxx、Hook 封装 | Hook 不暴露 setter、Hook 注释 |
| `state` | 状态管理 | 状态设计、数据流、状态恢复 | 表单状态恢复、优惠券状态 |
| `pattern` | 设计模式 | 设计模式、架构模式、代码组织 | 工厂模式、适配器模式 |
| `convention` | 编码规范 | 命名、格式、代码风格、注释 | 驼峰命名、Import 顺序 |
| `quality` | 代码质量 | 可维护性、性能优化、最佳实践 | 提前返回、避免过度拆分 |
| `workflow` | 工作流程 | 开发流程、调试方法、问题排查 | 数据刷新方式、错误处理 |

### Category 选择指南

**选择原则**：根据规则的**核心关注点**选择最匹配的 category

**决策树**：
```
规则涉及什么？
├─ API 调用/接口 → api
├─ TypeScript 类型 → types
├─ React 组件 → component
├─ 自定义 Hook → hook
├─ 状态管理/数据流 → state
├─ 设计模式/架构 → pattern
├─ 命名/格式/风格 → convention
├─ 代码质量/性能 → quality
└─ 开发流程/调试 → workflow
```

**冲突处理**：
- 如果规则同时涉及多个方面，选择**最核心**的那个
- 例如："API 工厂方法的命名规范" → 选 `api`（核心是 API）
- 例如："Hook 的类型定义" → 选 `hook`（核心是 Hook）

```markdown
**用户说了什么**：[用户的原始表达]

**为什么会这么说**：

- [分析：遗漏了什么重要信息]

✅ 好：Hook 不允许暴露内部 setter

## 相关记忆
- [相关记忆1](相关记忆1.md)

```

**Frontmatter 字段说明**：
- `category` - **必填**，从标准列表中选择唯一一个
- `priority` - **必填**，high/medium/low
- `tags` - **可选**，用于细粒度搜索的关键词数组
- `description` - **可选**，一句话描述

**反思记录说明**：
- 记录用户的原始反馈
- 分析为什么用户会提出这个要求
- 提取背后的通用原则
- 帮助 AI 理解记忆的深层含义

**文件命名示例**：
- `2026-03-10-hook-no-expose-setter.md`
- `2026-03-10-api-url-trailing-slash.md`
- `2026-03-10-component-props-required.md`
