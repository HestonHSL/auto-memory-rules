# Distillation Logic & Standards (提纯逻辑与标准)

> 本文件详细定义了如何将原始对话碎片转化为高质量长期记忆。

---

## 1. 意图识别表 (Intent Recognition)

在分析用户修改时，AI 必须对照下表判定意图，决定记录优先级。

| 意图类型       | 触发信号                         | 优先级 | 处理方式                      |
| :------------- | :------------------------------- | :----: | :---------------------------- |
| **偏好修正**   | 用户对风格调整（换库、改命名等） | 🔴 高  | 必须抽象后记录到 Layer 2      |
| **规范落地**   | 团队/公司强制规范（Lint/安全）   | 🔴 高  | 必须记录，标注"强制性"        |
| **功能纠偏**   | AI 误解需求，用户纠正逻辑        | 🟡 中  | 判定是否有模式可提取          |
| **业务补充**   | 用户补充了无法预知的业务上下文   | 🟡 中  | 记录于对应 `module-[name].md` |
| **一次性修改** | 明确说明"这里特殊处理"           | ⚪ 低  | 仅存 Layer 1 或不记录         |

---

## 2. 详细写入规范 (Writing Specifications)

### Layer 1：对话碎片 (.logs/)

- **原则**：忠实记录、包含上下文。
- **结构**：
  - `## [时间] [任务]`
  - `纠错/修改`: 描述具体变动。
  - `用户意图`: 基于上表分类。

### Layer 2：提纯规则 (.memory/)

- **原则**：抽象规则、指令化、高可检索性。
- **结构**：
  - `### [标题]`：包含技术关键词。
  - `场景描述`：何时适用。
  - `规则 (可执行)`：必须是能直接放入 Prompt 的指令。
  - `原因 / 强制性`：说明背后的逻辑和豁免权。

---

## 3. 质量门控 (Quality Gate Checklist)

写入 Layer 2 前必须通过这道检验：

- [ ] **抽象化**：是否去除了具体文件路径、具体行号、具体变量名？
- [ ] **可复用**：在下一个不相关的任务中，这条规则还能生效吗？
- [ ] **指令化**：如果你是另一个 AI，读了这条规则能立刻知道怎么做吗？
- [ ] **高价值**：这条规则是否解决了重复出现的纠错成本？

---

## 4. 冲突协议 (Conflict Protocol)

当新收集的记忆与旧记录冲突时：

1.  **判定类型**：是“规则演进”（旧的废弃）还是“场景特殊”（两者并存但范围不同）？
2.  **处理操作**：
    - 演进：覆盖旧条目，并修改日期。
    - 并存：重写旧条目，明确标注两个规则的不同适用场景/范围。
