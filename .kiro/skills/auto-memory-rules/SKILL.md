# Auto-Memory-Rules 核心执行准则

> 本文件是记忆管理的"唯一真理来源"。

## 核心价值

**自动学习用户偏好，形成个性化的代码规范 checklist**

1. **自动捕获** - 在对话中识别用户的偏好和纠错，实时记录到 fragments
2. **智能提纯** - 将重复出现的偏好提炼成可执行的规则
3. **主动应用** - 在后续任务中自动遵循这些规则

## 最高优先级原则

**用户的实时要求 > 历史记忆规则**

- 当用户的当前要求与历史记忆冲突时，无条件执行用户要求
- 将冲突视为"规则更新信号"
- 在对话结束时，**询问用户**是否要覆盖旧规则
- 记忆是参考，不是约束

## 任务生命周期

### 0. 对话内缓存

在同一对话中维护以下状态以减少 IO：
- `last_keywords` - 上次的关键词列表
- `last_topic_fingerprint` - 话题指纹（前 3 个关键词）
- `loaded_rules` - 本对话已读取的规则文件
- `pending_fragments` - 待提纯的碎片列表

### 1. 对话开始（Pre-task）

1. **检查目录**
   ```bash
   ls -la .kiro/skills/auto-memory-rules/memory/ 2>/dev/null
   ```

2. **自愈初始化**（如果目录不存在）
   - 创建 `memory/` 目录结构
   - 创建 `memory/index.md` 索引文件
   - 创建 `memory/rules/` 和 `memory/fragments/` 子目录

3. **加载相关记忆规则**
   - 根据用户请求的关键词，搜索相关规则
   - 将匹配的记忆规则注入到当前上下文
   - 如果没有找到相关记忆，明确说明"未找到相关历史记忆"

### 2. 对话执行（Execution）- 核心功能

**触发条件：**

| 意图类型 | 触发信号 | 优先级 | 处理策略 |
|---------|---------|--------|---------|
| **明确纠错** | "不对"、"错了"、"应该是..." | 🔴 高 | 立即反思 + 记录 |
| **要求修改** | "改成..."、"换成..."、"不要..." | 🔴 高 | 立即反思 + 记录 |
| **偏好表达** | "我更喜欢..."、"最好..."、"建议..." | 🟡 中 | 反思后记录 |
| **方法指导** | "用...命令"、"去...查询" | 🟡 中 | 反思后记录 |
| **交互优化** | 对回答方式、格式、风格的调整 | 🟡 中 | 反思后记录 |
| **一次性修改** | "这里特殊处理"、"临时..." | ⚪ 低 | **不记录** |

**自我反思流程：**

当用户提出任何修改要求时，AI 必须进行三层反思：

1. **表层反思**：用户说了什么？
   - 直接理解用户的字面要求

2. **深层反思**：为什么用户会这么说？
   - 我的回答哪里不够好？
   - 我遗漏了什么重要信息？
   - 我的方案有什么不合理的地方？
   - 我是否理解了用户的真实意图？

3. **原则提取**：这背后的通用原则是什么？
   - 是代码质量问题？（quality）
   - 是用户偏好？（convention）
   - 是最佳实践？（pattern）
   - 是项目规范？（workflow）
   - 是否可以泛化到类似场景？

**执行动作：**
1. **立即反思** - 理解用户要求的深层原因
2. **提取原则** - 识别可复用的通用规则
3. **创建 rule** - 记录到 `memory/rules/` 目录
4. **应用改进** - 在当前任务中立即应用

### 3. 对话结束（Settlement）

**话题切换判断：**

当出现以下任一条件，视为话题切换，触发结算：
- 明确转折词：「另外」「改成」「换成」「再来」「顺便」
- 本回合关键词与 `last_keywords` 差异 >= 40%
- 用户明确要求新增/修改规则
- 对话自然结束

**结算流程：**

1. **检查待提纯的碎片**
   ```bash
   ls -la .kiro/skills/auto-memory-rules/memory/fragments/
   ```

2. **执行记忆提纯**
   - 审查每个 fragment 的内容
   - 判断是否达到提纯标准（重复出现、明确偏好、可执行）
   - 检查是否已有相似规则（避免重复）
   - 将合格的 fragment 提纯为 rule 文件
   - 清理已提纯的 fragments

3. **更新索引和 Checklist**
   - 更新 `memory/index.md`
   - 自动生成 checklist：`.kiro/skills/auto-memory-rules/scripts/generate-checklist.sh`（输出到项目根目录 `CHECKLIST.md`）

**提纯判断标准：**
- ✅ 用户明确表达的偏好
- ✅ 重复出现的修改模式（2次以上）
- ✅ 可以抽象为通用规则
- ❌ 一次性的特殊情况
- ❌ 项目特定的实现细节

**质量门控 Checklist：**

核心问题：**这个规则下次能让用户少提修改要求吗？**

应该记录：
- ✅ 用户明确的偏好表达（"我更喜欢..."、"最好..."）
- ✅ 重复出现的修改模式（2次以上）
- ✅ 可抽象为通用原则（跨项目可用）
- ✅ 高成本的错误与修正路径
- ✅ 关键参数/设定/前置条件
- ✅ 用户对交互方式、回答格式的偏好
- ✅ 代码风格、架构模式的偏好

不应记录：
- ❌ 一次性的特殊处理
- ❌ 项目特定的业务逻辑
- ❌ 临时的解决方案
- ❌ 一问一答、没有可重用流程
- ❌ 纯概念解释（没有具体做法）

质量标准：
- [ ] **去具体化** - 移除文件路径、行号、具体变量名
- [ ] **高可执行性** - 规则必须是 AI 读完后能立刻执行的指令
- [ ] **跨任务通用** - 该规则在其他类似场景中依然有效
- [ ] **抗衰减** - 规则逻辑清晰，不会随时间推移而失效
- [ ] **理解原因** - 记录了"为什么"，不只是"做什么"

**冲突处理：**
- **规则演进** - 若用户明确推翻旧习惯，询问后覆盖旧规则
- **场景分支** - 若两个规则在不同场景下均有效，则合并并详细描述各自的触发场景

## 标准 Category 分类

创建 rule 文件时，必须从以下 category 中选择**唯一一个**：

| Category | 中文名称 | 适用场景 | 示例 |
|----------|---------|---------|------|
| `api` | API 相关 | API 调用、接口定义、HTTP 请求、URL 规范 | API URL 尾部斜杠、工厂方法 |
| `types` | 类型定义 | TypeScript 类型、接口、泛型、类型工具 | 查询参数类型、类型适配器 |
| `component` | 组件开发 | React 组件、JSX、组件设计、Props | 组件位置规则、Props 必选性 |
| `hook` | Hook 开发 | 自定义 Hook、useXxx、Hook 封装 | Hook 不暴露 setter、Hook 注释 |
| `state` | 状态管理 | 状态设计、数据流、状态恢复 | 表单状态恢复、优惠券状态 |
| `pattern` | 设计模式 | 设计模式、架构模式、代码组织 | 工厂模式、适配器模式 |
| `convention` | 编码规范 | 命名、格式、代码风格、注释 | 驼峰命名、Import 顺序 |
| `quality` | 代码质量 | 可维护性、性能优化、最佳实践 | 提前返回、避免过度拆分 |
| `workflow` | 工作流程 | 开发流程、调试方法、问题排查 | 数据刷新方式、错误处理 |

### Category 选择指南

**选择原则**：根据规则的**核心关注点**选择最匹配的 category

**决策树**：
```
规则涉及什么？
├─ API 调用/接口 → api
├─ TypeScript 类型 → types
├─ React 组件 → component
├─ 自定义 Hook → hook
├─ 状态管理/数据流 → state
├─ 设计模式/架构 → pattern
├─ 命名/格式/风格 → convention
├─ 代码质量/性能 → quality
└─ 开发流程/调试 → workflow
```

**冲突处理**：
- 如果规则同时涉及多个方面，选择**最核心**的那个
- 例如："API 工厂方法的命名规范" → 选 `api`（核心是 API）
- 例如："Hook 的类型定义" → 选 `hook`（核心是 Hook）

## 文件模板

### Fragment 模板

```markdown
# [关键词] 碎片记录

## 时间
YYYY-MM-DD HH:mm

## 场景
[用户在什么情况下提出的]

## 用户要求/纠错
[用户的原始表达]

## 执行结果
[按照用户要求执行的结果]

## 待确认
[是否需要更新规则？如何更新？]
```

### Rule 模板

```markdown
---
category: [从标准列表中选择唯一一个]
priority: [high|medium|low]
tags: [关键词1, 关键词2, 关键词3]
description: [一句话描述规则作用]
---

# [规则主题]

## 规则来源
用户反馈（YYYY-MM-DD）

## 触发场景
[用户在什么情况下提出的修改要求]

## 反思记录
**用户说了什么**：[用户的原始表达]

**为什么会这么说**：
- [分析：我的回答哪里不够好]
- [分析：遗漏了什么重要信息]
- [分析：方案有什么不合理的地方]

**提取的原则**：[这背后的通用原则是什么]

## ⚠️ 重要提示
当用户明确要求与本规则冲突时，优先执行用户的实时要求。

## 核心原则
[一句话概括规则]

## 具体要求
- [可执行的指令1]
- [可执行的指令2]

## 检查点
- [ ] [规则陈述1：描述应该做什么]
- [ ] [规则陈述2：描述应该做什么]

注意：检查点使用陈述句，不是疑问句或错误描述
✅ 好：Hook 不允许暴露内部 setter
❌ 差：Hook 是否暴露了 setter？
❌ 差：Hook 暴露了 setter

## 优先级说明
🔴 高优先级 - [说明为什么是高优先级]
🟡 中优先级 - [说明为什么是中优先级]
🟢 低优先级 - [说明为什么是低优先级]

## 示例

### ✅ 好的示例
[示例代码或描述]

### ❌ 避免的示例
[反例]

## 相关规则
- [相关规则1](相关规则1.md)
- [相关规则2](相关规则2.md)
```

**Frontmatter 字段说明**：
- `category` - **必填**，从标准列表中选择唯一一个
- `priority` - **必填**，high/medium/low
- `tags` - **可选**，用于细粒度搜索的关键词数组
- `description` - **可选**，一句话描述

**反思记录说明**：
- 记录用户的原始反馈
- 分析为什么用户会提出这个要求
- 提取背后的通用原则
- 帮助 AI 理解规则的深层含义

## 代码审查流程

### 生成 Checklist

**自动生成时机：**
- 对话结束时，如果有新的 rule 被创建或修改

**手动生成：**
```bash
.kiro/skills/auto-memory-rules/scripts/generate-checklist.sh
```

### 触发审查

在对话中主动请求：
```
"用 checklist 审查代码"
"根据我的偏好规则检查这个文件"
```

AI 会：
1. 加载项目根目录的 `CHECKLIST.md`
2. 逐项检查代码
3. 报告所有问题
4. 询问是否自动修复

### 审查输出格式

```markdown
# 代码审查报告

## 文件: src/components/FileList.tsx

### ❌ 发现 3 个问题

#### 1. Hook 暴露了内部 setter（第 15 行）
- 规则: hook-no-expose-setter.md
- 代码: `return { fileList, setFileList };`
- 建议: 提供语义化方法如 updateFileItem
- 优先级: 🟡 中

### ✅ 符合规范（5 项）
- Hook 函数有 JSDoc 注释
- Import 顺序正确

---

是否自动修复以上问题？[Y/n]
```
