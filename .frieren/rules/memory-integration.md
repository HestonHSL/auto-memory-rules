# 记忆系统集成提醒

## 核心目标
**捕获开发者对 AI 生成代码的纠错和修改要求，积累错误模式，形成自检清单，避免重复犯错**

## 关键触发信号

### 立即记录（高优先级）
- ❗ **明确纠错**："不对"、"错了"、"应该是..."
- ❗ **要求修改**："改成..."、"换成..."、"不要..."
- ❗ **指出遗漏**："还要..."、"别忘了..."、"漏了..."
- ❗ **偏好表达**："我更喜欢..."、"最好..."、"建议..."

### 评估后记录（中优先级）
- ⚠️ **方法指导**："用...命令"、"去...查询"、"应该查看..."
- ⚠️ **经验分享**："例如..."、"一般..."、"通常..."
- ⚠️ **流程建议**："查不到的时候..."、"像...这种情况..."
- ⚠️ **交互优化**：用户对回答方式、展示格式、沟通风格的调整

## 自我反思机制

当用户提出任何修改要求时，AI 应该立即进行批判性反思：

### 反思三问
1. **为什么用户会提出这个要求？**
   - 我的回答哪里不够好？
   - 我遗漏了什么重要信息？
   - 我的方案有什么不合理的地方？

2. **这个要求背后的原则是什么？**
   - 是代码质量问题？
   - 是用户偏好？
   - 是最佳实践？
   - 是项目规范？

3. **这个原则可以泛化吗？**
   - 是否适用于类似场景？
   - 是否值得记录为规则？
   - 优先级应该是多少？

## 工作流程

### 对话开始
1. 加载 `.frieren/skills/auto-memory-rules/memory/rules/` 中的历史规则
2. 加载 `.frieren/skills/auto-memory-rules/memory/CHECKLIST.md` 查看结构化清单
3. 根据任务关键词搜索相关规则
4. 形成"避坑清单"

### 对话执行
1. **持续监听**用户的所有反馈信号（纠错、修改、偏好、优化）
2. **自我反思**：
   - 为什么用户会提出这个要求？
   - 我的回答哪里不合理？
   - 这背后的原则是什么？
3. **批判性思考**：是否满足记录标准？
   - 可复用性、全局性、特殊性、经验性、重复性
4. **立即记录**：创建 rule 文件到 `rules/` 目录

### 对话结束
1. 统计本次记录的 rule 数量
2. 运行 scripts 生成索引和清单：
   ```bash
   bash .frieren/skills/auto-memory-rules/scripts/generate-index.sh
   bash .frieren/skills/auto-memory-rules/scripts/generate-checklist.sh
   ```

## Memory 目录结构

```
.frieren/skills/auto-memory-rules/memory/
├── rules/                    # 所有规则（AI 负责创建）
│   ├── 2026-03-09-xxx.md
│   └── ...
├── CHECKLIST.md             # 自动生成（generate-checklist.sh 负责）
└── index.md                 # 自动生成（generate-index.sh 负责）
```

**维护职责**：
- **AI** - 创建 rule 文件到 `rules/` 目录
- **generate-index.sh** - 自动生成 `index.md`（分类索引）
- **generate-checklist.sh** - 自动生成 `CHECKLIST.md`（自检清单）
- **开发者** - 手动维护 `README.md`（如需要）

## 核心原则

1. **用户当前请求 > 历史规则** - 冲突时优先执行用户请求，但要提醒
2. **每次反馈都反思** - 不只是纠错，任何修改要求都要思考原因
3. **批判性反思 = 主动避错** - 理解用户要求背后的原则，主动规避
4. **持续改进 = 错误递减** - 目标是让同类问题越来越少
5. **自我学习 = 理解偏好** - 不只记录规则，更要理解用户的思维方式

## 多维度思维

在所有阶段应用：
- **系统思维** - 分析整体架构和依赖关系
- **辩证思维** - 评估方案优劣
- **创新思维** - 探索更优解决方案
- **批判思维** - 验证实现正确性
